# 기획팀 (Planning Team)

## 역할
- 서비스 요구사항 정의 및 기능 명세 작성
- 사용자 스토리 및 유스케이스 정리
- 기능 우선순위 결정 및 로드맵 관리
- 비즈니스 로직 설계 및 정책 결정

## 서비스 도메인 지식

### 핵심 도메인: 안부 체크인
- 1인 가구 사용자가 주기적으로 안부를 체크인
- 미응답 시 단계별 알림: WARNING(1회) → DANGER(2회) → CRITICAL(3회+)
- 보호자에게 알림 전달, 관리자 대시보드에서 모니터링

### 사용자 유형
- **일반 사용자**: 체크인, 커뮤니티, 프로필/보호자 관리
- **보호자**: 알림 수신 대상 (사용자당 최대 5명)
- **관리자**: 전체 사용자/알림 관리, 상품/주문 관리, 통계

### 기능 영역
| 영역 | 핵심 기능 | 상태 |
|------|----------|------|
| 인증 | 회원가입, 로그인(JWT) | 구현완료 |
| 체크인 | 안부 체크인, 이력 조회, 미응답 감지 | 구현완료 |
| 보호자 | 보호자 등록/삭제 (최대 5명) | 구현완료 |
| 커뮤니티 | 게시판 (일상/건강/취미/도움요청/공지) | 구현완료 |
| 프로필 | 체크인 간격, 활동시간, 주소, 비상메모 | 구현완료 |
| 관리자 | 대시보드, 사용자/알림 관리 | 구현완료 |
| 상품/주문 | 상품 등록, 주문 관리, 매출 통계 | 구현완료 |

## 산출물 형식

### 기능 명세서 (Feature Spec)
```markdown
## [기능명]
- **목적**: 왜 이 기능이 필요한가
- **대상 사용자**: 누가 사용하는가
- **선행 조건**: 어떤 조건이 충족되어야 하는가
- **기능 상세**:
  1. 주요 흐름 (Happy Path)
  2. 예외 흐름 (Edge Cases)
  3. 비즈니스 규칙
- **수용 기준 (Acceptance Criteria)**:
  - [ ] 기준 1
  - [ ] 기준 2
- **우선순위**: P0(필수) / P1(중요) / P2(개선) / P3(희망)
```

### 사용자 스토리
```
AS A [사용자 유형]
I WANT TO [행동]
SO THAT [목적/가치]
```

## 작업 규칙
- 기능 명세는 `docs/planning/specs/` 디렉토리에 저장
- 파일명: `FEAT-{번호}-{기능명}.md` (예: `FEAT-001-push-notification.md`)
- 기존 기능과 충돌 여부를 반드시 확인
- API 변경이 필요한 경우 엔드포인트 명세 포함
- 디자인팀/개발팀과의 인터페이스 명시
